# Derive Macro Indexing Implementation

## Overview

This implementation adds support for indexing impl blocks generated by `#[derive(...)]` macros in the rust-brain code intelligence platform.

## Problem Statement (GAP-005)

Functions generated by macros (like derive macros: Debug, Clone, Serialize, etc.) were not captured in the index. Only source-level items were tracked.

## Solution

### 1. New Module: `derive_detector.rs`

Created a new module that:
- Parses original source to find structs/enums with `#[derive(...)]` attributes
- Parses expanded source to find all impl blocks
- Matches impl blocks with derive attributes to identify generated impls
- Returns a map of impl signatures to derive sources

### 2. Data Model Changes

#### `ParsedItem` (parsers/mod.rs)
Added `generated_by: Option<String>` field to track macro-generated items.

#### `ParsedItemInfo` (pipeline/mod.rs)
Added corresponding `generated_by` field for state tracking.

#### Database Schema (init-db.sql)
Added `generated_by TEXT` column to `extracted_items` table with an index.

### 3. Pipeline Integration

#### ParseStage (pipeline/stages.rs)
- Instantiates `DeriveDetector` alongside `DualParser`
- After parsing, runs derive detection on original vs expanded source
- Annotates impl items with `generated_by` metadata

#### ExtractStage (pipeline/stages.rs)
- Stores `generated_by` in the database when extracting items

#### GraphStage (pipeline/stages.rs)
- Includes `generated_by` as a node property in Neo4j

#### EmbedStage (pipeline/stages.rs)
- Preserves `generated_by` through the embedding pipeline

### 4. API Enhancements

Added helper methods to `PayloadValue` in qdrant_client.rs:
- `as_str()` - Get value as string
- `as_i64()` - Get value as integer
- `as_bool()` - Get value as boolean

## Usage Examples

### Query for derive-generated items
```sql
SELECT fqn, name, generated_by 
FROM extracted_items 
WHERE generated_by IS NOT NULL;
```

### Query for specific derive type
```sql
SELECT fqn, name 
FROM extracted_items 
WHERE generated_by = 'derive(Debug)';
```

### Neo4j query for derive-generated impls
```cypher
MATCH (n:Impl {generated_by: 'derive(Clone)'})
RETURN n.name, n.fqn
```

## Test Results

All 4 derive detector tests pass:
- `test_find_derives` - Detects `#[derive(...)]` attributes
- `test_find_impl_blocks` - Extracts impl blocks from expanded source
- `test_match_derives_to_impls` - Matches derives to generated impls
- `test_no_match_for_manual_impl` - Doesn't falsely identify manual impls

## Limitations

1. **Pattern matching** - Uses regex-based matching which may not handle all edge cases (e.g., complex generic types in derives)

2. **Trait path resolution** - Currently extracts only the trait name, not the full path (e.g., `serde::Serialize` becomes `Serialize`)

3. **Proc macro expansion** - Only handles standard derives; custom proc macros are not detected

## Future Improvements

1. Use syn for more robust derive attribute parsing
2. Support custom derive macro detection via attribute tracking
3. Add support for other macro-generated items (e.g., function-like macros)
4. Track macro expansion relationships in Neo4j graph
